# Compound engineering

The single most useful thing you can install after the first four chapters is the **Compound Engineering plugin** by Avery from Every. It packages the full `brainstorm → plan → work → review → compound` loop as slash commands you can run against any codebase.

This chapter is how you install it, use it, and why the loop compounds.

## The loop in five commands

- `/brainstorm` — widen the problem before you narrow it
- `/plan` — decide what to build and in what order, written down
- `/work` — execute the plan, one atomic change at a time
- `/review` — second pass on the output, against the plan
- `/compound` — write down what you learned so next time is cheaper

<Diagram chart={"flowchart LR\n    B[brainstorm] --> P[plan]\n    P --> W[work]\n    W --> R[review]\n    R --> C[compound]\n    C -.->|next problem| B"} caption="Each pass leaves documents the next brainstorm and plan will draw on." />

Most dev work fits this shape. The plugin makes the shape visible — and reusable.

## Install

Two steps from inside Claude Code:

```
/plugin marketplace add EveryInc/compound-engineering-plugin
/plugin install compound-engineering
```

The first command registers the marketplace; the second installs the plugin from it. If you prefer a direct source, clone the [Compound Engineering repo](https://github.com/EveryInc/compound-engineering-plugin) and point Claude at the local path.

Verify:

```
/plan --help
```

If the command is there, you're in.

## Use it on something real

Pick a task you'd work on this week. Not a toy — something with enough shape to plan. A small feature, a migration, a performance issue. This is what the loop is for.

**1. Brainstorm.** Start with:

```
/brainstorm: I want to add rate limiting to our public API
```

Don't skip to the solution. Let Claude widen the problem — what traffic patterns, what attacker model, what cost. Most of the value is in the questions it asks you back.

**2. Plan.** When the problem is clear:

```
/plan
```

Claude writes a task list with dependencies, saves it to a markdown file under `docs/plans/`, and stops. Read it. Edit it. Accept only when you'd hand it to a teammate without blushing.

**3. Work.** Against the approved plan:

```
/work
```

Claude ticks tasks off one at a time, commits small, doesn't batch. If something surprises you in the diff, stop and update the plan — don't power through.

**4. Review.** When work is done:

```
/review
```

Claude reads its own output against the plan and surfaces what's missing, what drifted, what needs a second look. This catches ~70% of the "it compiles but…" class of problems.

**5. Compound.** Finally:

```
/compound
```

Claude writes a short doc to `docs/solutions/` with the problem, the fix, and what was non-obvious. These docs become the memory your next `/brainstorm` and `/plan` will draw on.

## Why the loop compounds

The first time through, you save an hour. Linear.

The second time, `/compound` has left docs in your repo. Your next `/brainstorm` is faster because the unknowns shrunk. Your next `/plan` references the prior one.

The tenth time, half of your planning is already written. The skills you've extracted handle routine work. You're spending time on new problems instead of rediscovering old ones. That's the compound.

## What to try this week

1. Install the plugin
2. Pick one real task
3. Run the full loop — all five commands — even if it feels slow
4. Do it again on a second task
5. Notice what's already written after `/compound`

Two full loops is enough to feel the shape. After that, the loop runs you.

## Build a skill — the agentic way

You just used someone else's plugin to structure your work. Now turn that into a habit for yourself: when something repeats, extract a skill. Don't hand-write the `SKILL.md`. Have Claude do it.

### Two meta-skills that write skills

Both do the same job: guide you through discovery, draft a well-shaped `SKILL.md`, write it to disk. Pick one.

- **`skill-creator`** — shipped by Anthropic in the [`anthropics/skills`](https://github.com/anthropics/skills) repo. Install via `/plugin` from that marketplace. Canonical source, minimal, teaches the shape.
- **`craft-skill`** — part of the [Heart of Gold toolkit](https://github.com/ondrej-svec/heart-of-gold-toolkit). Opinionated, uses a five-phase workflow with a technique checklist, produces higher-conformance output by default.

### The walk-through

With either installed, in a real project, run:

```
/skill-creator
```

(or `/craft-skill`, depending on which you installed).

Claude will ask, one question at a time:

1. **What should this skill do?** A sentence. *"Write a release note in our voice from a list of merged PRs."*
2. **When should it activate?** Trigger phrases. *"When the user asks for a release note, changelog, or ship note."*
3. **Who's the user?** Specific. *"Me and the engineering team."*
4. **What should success look like?** The output shape. *"A 100-200 word note with three sections: what shipped, what to watch, what's next."*

Answer each one as specifically as you can. Vague answers make vague skills.

Claude then drafts the `SKILL.md` — frontmatter (`name`, `description`, `when_to_use`), phases, examples, anti-patterns. It shows you the diff. You review, tweak, accept.

The file lands in `~/.claude/skills/my-skill/SKILL.md`. Next session, when you ask for a release note, the skill activates automatically.

### What Claude produced

The frontmatter will look roughly like this — this is the shape Claude writes for you, not a template you fill in:

```md
---
name: release-note
description: Write release notes in our team's voice from merged PRs.
when_to_use: When asked for a release note, changelog, or ship note.
---

# Release note

When the user asks for a release note:

1. Ask for the PR range or list if not given.
2. Read each PR title + description.
3. Group into: what shipped · what to watch · what's next.
4. Write 100–200 words in the voice at `docs/voice.md`.
```

Skills live under 500 lines. Longer content goes into bundled files: `references/` for reading material, `scripts/` for helpers, `assets/` for templates. This is called *progressive disclosure* — the metadata is always loaded, the body loads when triggered, the extras load only if needed.

### Agent Skills open format

The format is [an open spec](https://agentskills.io) — frontmatter fields, loader behaviour, progressive-disclosure rules. Anthropic's full reference is in [anthropics/skills/spec/agent-skills-spec.md](https://github.com/anthropics/skills/blob/main/spec/agent-skills-spec.md). Skills you write work in Claude Code today and will work in any tool that adopts the spec tomorrow.

### The real endgame

Do this three or four times — extract a skill whenever something repeats — and suddenly you have a small collection. Bundle them with a `.claude-plugin/plugin.json`, publish to GitHub, and that's **your plugin**. Same shape as Compound Engineering, just yours. Your team installs once and inherits everything you've learned to systematise.

That's compound engineering, all the way down.
