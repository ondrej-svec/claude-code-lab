# Iteration and control

The first prompt is never the last prompt. The value in Claude Code isn't the output — it's the loop that gets you to the right output in three or four rounds instead of one frustrated rewrite.

## Rewind

Double-tap **Esc** in the Claude window. You're back one message. File changes are rolled back. The conversation branches. Same effect as the `/rewind` slash command, just faster.

This is the single most useful keyboard shortcut in the whole tool. Use it whenever:

- Claude went down the wrong path and you want to retry the prompt
- You spotted a better way to phrase the ask mid-thought
- You want to branch an experiment without committing to it
- The diff surprised you and you'd rather not have it

Rewind is cheap. Re-prompting from scratch is expensive. Use the shortcut. Chapter 8.2 has the full keybinding reference.

## Narrow the scope

The fastest way to bad output is a prompt that asks for too much at once.

**Wide:**
> Refactor this file to use dependency injection, add tests, and migrate it to TypeScript strict mode.

**Narrow, three prompts:**
> Refactor this file to use dependency injection. Don't touch tests or types yet.

> Now add tests for the refactored interface.

> Now enable strict mode and fix what breaks.

Three narrow prompts land three times faster than one wide one. Each one gets a clean review.

## "That's not quite right"

When Claude does something close but off, resist the urge to rewrite from scratch. Describe the gap:

> The helper function is fine, but move it into `utils/time.ts` — we keep time-related helpers there.

Claude will move it. The structure holds. You didn't have to re-explain the whole task.

## When to start fresh — `/clear` vs `/compact`

Context windows are large but not infinite. Two commands, different tools:

- **`/compact`** — same conversation, Claude summarises the history so far and throws the raw turns away. Use when you're mid-task and need breathing room.
- **`/clear`** — new conversation, old one archived (resumable via `/resume`). Use when you've pivoted to a different area of the codebase, or the conversation has 30+ turns and is drifting.

A clean conversation with good `CLAUDE.md` is almost always faster than a long one.

## Read the diff before accepting

This is the single habit that separates people who like working with agents from people who don't trust them.

Before hitting "accept":

- Scroll through the diff — all of it
- Check for unintended changes to unrelated files
- Check for new dependencies (did it add a library you don't need?)
- Check for tests changing in ways you didn't ask for

Ten seconds of review saves an hour of debugging later.

## Plan mode

Press **Shift+Tab** to cycle permission modes:

1. **default** — Claude asks before editing
2. **acceptEdits** — Claude edits freely, you review diffs after
3. **plan** — Claude can read, think, and propose a plan, but **cannot edit or run anything** until you leave plan mode
4. **auto** (on some plans) — Claude edits and runs without prompting

Plan mode is the one most worth knowing. In it, Claude reads the codebase, thinks through the task, writes a plan to `docs/plans/` (or wherever your convention lives), and stops. You review. If the plan is right, you hit Shift+Tab to leave plan mode and `/work` against it. If it's wrong, you fix it or rewind.

Use plan mode for:

- Multi-file changes where you want to see the shape before commits happen
- Anything with a destructive step (migrations, deletes, rewrites)
- Tasks where the "what" matters as much as the "how"
- Any time the Compound Engineering loop (chapter 6) uses `/plan`

Don't use plan mode for single-line fixes or one-function refactors — the overhead isn't worth it.
